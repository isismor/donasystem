Comparativo02
ChatGPT, Claude, Gemini vs OpenClaw
ChatGPT

Chat genérico. Sem memória persistente. Sem automação. Sem identidade. Responde perguntas, mas não trabalha para você.

Melhor raciocínio. Projetos. Mas sem integração com canais, sem crons, sem agentes autônomos, sem orquestração.

Integração com Google. Bom para pesquisa. Sem servidor, sem autonomia, sem sistema operacional próprio.

OpenClaw

Sistema completo. Servidor próprio. Memória permanente. Crons. Skills. Multi-agente. Canais integrados. Autonomia real. Trabalha 24/7.

A diferença: os outros são ferramentas que você usa. OpenClaw é um sistema que trabalha para você.

Higiene08
Higiene do sistema
Consolidação de memória

Notas diárias são consolidadas nos topic files periodicamente. Informação não se acumula sem organização. Cron automático à noite.

Limpeza de lições táticas

Lições com mais de 30 dias são revisadas e deletadas se não forem mais relevantes. Estratégicas são permanentes.

Git workspace

Todas as mudanças comitadas regularmente. Workspace limpo = sistema previsível. Boot check alerta se tem arquivos sujos.

Indexação de memória

Comando openclaw memory index atualiza o índice semântico. Buscas por contexto, não por palavra-chave.

Revisão semanal

Toda segunda-feira: revisar projetos ativos, consolidar notas, atualizar status, deletar lições vencidas.

Regra inviolável

Antes de qualquer compactação de contexto: extrair decisões e lições primeiro. Nunca perder informação.

Crons e Heartbeats
O que são?

Crons: tarefas agendadas que rodam automaticamente. O agente trabalha enquanto você dorme.

Heartbeats: pulso periódico que verifica a saúde do sistema. Se algo está errado, alerta imediatamente.

Heartbeat checklist
  • ✓ Há pendências com mais de 3 dias?
  • ✓ Projetos parados há mais de 7 dias?
  • ✓ Algum cron falhou?
  • ✓ Nota diária do dia foi criada?
  • ✓ Notas antigas para consolidar?

Arquitetura11
Arquitetura de agentes
Hub central + tópicos

Uma única instância do OpenClaw assume identidades diferentes conforme o tópico do Telegram. Cada agente tem seu espaço, personalidade e papel. Tópico determina quem responde.

Como funciona
  • 1 gateway, 1 modelo, 10 identidades
  • Cada agente tem soul, identity, knowledge próprios
  • Orquestração via sessions_spawn (sub-agentes)
  • Handoffs via completion events entre agentes
  • Estado sempre gravado em projects.md

PDF Pipeline14
Framework de geração de PDF
Pipeline unificado

Todos os agentes seguem o mesmo processo. Qualquer agente que precise gerar PDF usa o mesmo pipeline, template e design system.

1. Conteúdo (Masterson ou outro agente)
↓
2. HTML com Template C + Design System ISIS v7
↓
3. Conversão para PDF via Puppeteer no servidor
↓
4. Entrega para Isis
Design System ISIS v7
  • Paleta: Marfim, Creme, Cacau, Ouro, Mirror
  • Fontes: DM Serif Text, DM Sans, IBM Plex Mono
  • Template C "Bold Statement" (aprovado nota 9/10)
  • Containers com regra de cor unificada por página
Regras invioláveis
  • height: 297mm fixo por página (nunca min-height)
  • Conteúdo distribuído manualmente por página
  • Google Fonts sempre no head
  • Puppeteer local (fallback: YakPDF via RapidAPI)

Atlas17
A importância do Atlas
Estrategista e Orquestrador

Atlas não executa. Atlas planeja, mapeia riscos e coordena cadeias de produção com 2 ou mais agentes em sequência. Lê documentações antes de agir.

Por que ele existe?
  • Sem Atlas: cada agente trabalha isolado, sem visão do todo
  • Com Atlas: cadeias coordenadas com gates de aprovação
  • Resultado: output final passa por múltiplos especialistas antes de sair
O que ele faz na prática
  • Lê documentações e mapeia riscos antes de executar
  • Propõe planos estruturados com sequência de agentes
  • Spawna cada step e monitora completion events
  • Atualiza projects.md a cada step concluído
  • Se cadeia travar: retry 2x, depois alerta Isis
Regra de ouro

Gates verdes (entre agentes): segue automático. Gates vermelhos (output público): para e espera ok de Isis.
